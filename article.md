# 小さなAIに「考える力」を授ける ── オンデバイスLLMの推論精度を2倍にした全記録

## プロローグ：3行のコードで始まる挫折

iOS 26で、Appleは `FoundationModels` フレームワークを公開した。iPhoneやMacに内蔵された言語モデルに、Swiftコードで直接話しかけられる。

```swift
let session = LanguageModelSession()
let response = try await session.respond(to: "What is 7 x 8?")
// → "56"
```

クラウド不要。APIキー不要。遅延ゼロ。月額無料。素晴らしい。

ところが、少し込み入った質問をすると──

```
Q: "太郎は時速4kmで歩いている。1時間後に次郎が時速6kmで追いかけた。
    太郎が出発して何時間後に追いつく？"

A: "2/3時間後です"
```

**間違っている。** しかも自信満々に。正解は3時間後だ。

同じモデルに、同じ問題を5回聞くと、5回とも違う間違い方をする。ある時は2/3、ある時は5/3、ある時は2。安定して間違える。

ChatGPTなら一発で解ける。でもChatGPTはクラウドに依存し、月額$20かかり、オフラインでは使えない。

ここから、1つの問いが始まった。

**「モデルを大きくできないなら、考え方を外から教えればいい」**

この記事は、3,600行のSwiftコードで小さなオンデバイスモデルの推論精度を40%から90%に引き上げ、最終的にはあの追いつき算すら正解させるに至った、試行錯誤の全記録だ。

## 第1章：7本のパイプライン ── 「たくさん考えれば賢くなる」仮説

ChatGPTの「o1」モデルが注目を集めた時、世界中の開発者が同じことを考えた。

*「思考の連鎖（Chain of Thought）を外部パイプラインとして組めば、小さなモデルでも推論能力を底上げできるのでは？」*

そこで、7種類の「考え方のフレームワーク」を設計した。

```
Direct（直接）:       質問 → 回答                     1回の推論
Sequential（逐次）:   分析 → 回答                     2回
CritiqueLoop（批評）: 回答 → 批評 → 修正              3回
Rethink（再考）:      言い換え → 回答 → 検証           3回
BranchMerge（分岐）:  分析 → 3並列回答 → 統合         6回
SelfConsistency:      3独立回答 → 多数決              3回
StepByStep:           3独立回答 → 多数決（別方式）     3回
```

人間で言えば、Directは「即答」、Sequentialは「考えてから答える」、CritiqueLoopは「答えてから見直す」、BranchMergeは「3人に聞いて統合する」。

推論回数が多いほど、より深く考えているはずだ。直感的にはそう思える。

**この直感は、完全に裏切られた。**

---

## 第2章：衝撃 ── ステップを増やすほど馬鹿になる

10問のベンチマークを用意した。条件分岐、変数追跡、カードのスワップなど、「1回で解くと間違えやすい」問題ばかりだ。

全パイプラインの成績を並べた瞬間、目を疑った。

| パイプライン | 推論回数 | 正答率 |
|:-----------|:------:|:-----:|
| Direct | 1回 | 40% |
| **Sequential** | **2回** | **70%** |
| CritiqueLoop | 3回 | 50% |
| Rethink | 3回 | 70% |
| StepByStep | 3回 | 60% |
| BranchMerge | 6回 | 50% |

**6回も推論するBranchMergeが、何もしないDirectと同じ50%。**

なぜか。

### エラーは雪だるまのように膨らむ

小さなモデルの1回の推論には、そこそこのエラー率がある。問題は、パイプラインの各段階が前段階の出力に依存していることだ。

1段階目が間違えると、2段階目はその間違いを前提に推論する。3段階目は、間違いの上に間違いを重ねる。6段階のBranchMergeでは、Analyze（分析）段階が立てた誤った前提が、3つの並列ブランチすべてに伝染し、Merge（統合）段階がそれをもっともらしく1つにまとめ上げ、もっともらしい不正解が完成する。

**ステップを増やすことは「考えを深める」のではなく「間違える機会を増やす」ことだった。**

グラフにすると、推論回数と正答率の関係は右肩下がりの曲線を描く。唯一の例外が、2回の推論だった。

## 第3章：2段階のスイートスポット

データは「2段階が最適」と叫んでいた。

では、その貴重な2回で何をすべきか？ 3つの戦略を同じ条件で比較した。

```
Sequential:    分析 → 解答        「考えてから解く」
CritiqueLoop:  解答 → 批評+修正   「解いてから直す」
Rethink:       解答 → 独立検証     「解いてから他人にチェックしてもらう」
```

### 自分で自分をチェックすると、何が起きるか

CritiqueLoop（解いてから直す）は、直感的には最も合理的に見える。人間のコードレビューと同じ発想だ。

しかし、ベンチマークの結果は残酷だった。

```
Q: "農夫が羊を17匹持っている。9匹以外が全部逃げた。何匹残っている？"

[Solve]    17 - 9 = 8匹。
[Critique] 計算を確認。17 - 9 = 8。問題なし。
[Final]    答え：8匹

                        ← 正解は9匹
```

「all but 9（9匹を除いて全部）」は「9匹が残る」という意味だ。Solve段階が問題文を誤読し、Critique段階はそれを素通りさせた。

これは**確証バイアス**だ。間違った答えを目の前に置かれると、人間もAIも、それが正しい理由を探してしまう。

### 「他人の目」がバイアスを破る

Rethinkパイプラインは異なるアプローチを取った。

Solve段階とVerify段階で**完全に独立したセッション**を使う。Verify段階はSolveの思考過程を一切見ない。問題文と最終的な回答だけを受け取り、**自分でゼロから解き直す**。

```
[Solve]   17 - 9 = 8。答え：8
[Verify]  問題をゼロから解く。
          「all but 9」= 9匹が残る。
          提案された8は間違い。答え：9 ✓
```

同じモデル、同じパラメータ。なのに、新鮮なセッションで問題文だけを見せると正しく読み取れる。Solveの思考過程という「先入観」がないからだ。

これは論文の査読プロセスと同じ構造だ。著者（Solve）と査読者（Verify）は独立していなければ意味がない。

## 第4章：セッション分離 ── 見えない壁を壊す

2段階が最適だとわかった。独立検証が最も効果的だとわかった。だが、もう1つ重要な発見があった。

**マルチターンセッション（同一セッション内の会話）は、独立セッションに劣る。**

初期実装のSequentialパイプラインでは、1つのLanguageModelSessionで2ターン分の会話を行っていた。

```swift
// 旧: 1つのセッションで会話
let session = LanguageModelSession(instructions: "...")
let think = session.respond(to: "まず分析して")  // Turn 1
let solve = session.respond(to: "次に解答して")  // Turn 2
```

これを、各段階で別々のセッションに変更した。

```swift
// 新: 段階ごとに独立セッション
let think = provider.generate(system: "分析者", user: "問題を分析して")
let solve = provider.generate(system: "解答者", user: "問題文 + 分析結果の要約")
```

**この変更だけで正答率が60%から70%に改善した。**

原因は「コンテキスト汚染」だ。

マルチターンセッションでは、Think段階の冗長な出力がセッションのコンテキスト窓に残り続ける。Apple Intelligenceモデルのコンテキスト窓は約1,200文字。Thinkの出力が500文字を食えば、Solveに残されたスペースは700文字しかない。窮屈な部屋で考えるようなものだ。

独立セッションでは、Solve段階はThinkの**要約だけ**を受け取る。広い部屋で、必要な情報だけを手元に置いて考えられる。

## 第5章：一語の追加が精度を変える

パイプライン構造の最適化が一段落した後、次の戦場はプロンプトだった。

### 反復回数を数え間違える問題

```
Q: "7から始めて、偶数なら2で割り、奇数なら2倍にする。6回繰り返すと？"

正解: 7→14→7→14→7→14→7（答え：7）

全パイプライン共通の回答: 14（5回で止まる）
```

どのパイプラインでも、「6回繰り返す」指示を5回で打ち切る。

### 「紙に書け」と言えばいい

システムプロンプトに1行追加した。

```
追加: "After EACH step, write State: [current values]."
```

モデルに「State: 14」「Iteration 3:」と**テキストとして明示的に書かせる**ことで、暗黙のカウントを排除した。

結果：この問題の正答率が**0%から100%**になった。

小さなモデルは「頭の中で状態を保持する」のが苦手だ。だが「紙に書く」ことはできる。生成したテキスト自体が、モデルにとっての外部メモリになる。

## 第6章：「まずアプローチを考えろ」── 追いつき算の突破

ベンチマークの正答率は90%に達した。しかし、冒頭の追いつき算はまだ解けなかった。

```
Q: "太郎は時速4km、次郎は1時間後に時速6kmで追いかける。何時間後に追いつく？"
A: "4 ÷ 6 = 2/3時間後"  ← 不正解。太郎がまだ歩いていることを無視
```

ベンチマーク問題との違いは明白だった。ベンチマークは「Step 1: 2倍にしろ、Step 2: 10より大きければ...」と**手順が明示されている**。追いつき算は**自分で解法を見つける必要がある**。

「相対速度を使う」という概念に、モデルが自力で辿り着けない。

### 一語の追加

```
追加前: "Think carefully before answering."
追加後: "Before solving, first identify what approach, formula, or concept is needed."
```

この一文を加えた結果──

```
[Solve]
1. 太郎の距離: 4km/h × 1h = 4km
2. 次郎との速度差: 6 - 4 = 2km/h   ← 相対速度を特定！
3. 追いつく時間: 4km ÷ 2km/h = 2h
4. 太郎の出発から: 1 + 2 = 3時間後 ✓
```

**正解。** 「まずアプローチを特定せよ」という指示が、モデルに「速度差」という概念を引き出させた。

「考えろ」では漠然としすぎる。「何を使って考えるか、まず決めろ」と言えば、モデルは道具箱から正しい道具を選べることがある。

## 第7章：7つから3つへ ── 削ぎ落としの設計

全ての実験を経て、7つのパイプラインは3つに集約された。

```
Direct:    単純な質問（1回のLLM呼び出し、高速）
Rethink:   推論が必要な質問（2回、高精度）
Verified:  制約充足問題（LLMで条件抽出 → 決定論的ソルバー）
```

**削除したコードは1,600行。** 残ったのは、データが「効く」と証明したものだけだ。

Rethinkパイプラインの最終形は、驚くほどシンプルだ。

```
Stage 1 [独立セッションA]:
  「アプローチを特定し、ステップごとに状態を記録しながら解け」

Stage 2 [独立セッションB]:
  「この下書きをレビューして改善せよ」
  → 拒否された場合のみ Stage 1 の出力を採用
```

普段は2回のLLM呼び出しで完結する。追加したコードはわずか150行。モデルの再学習は不要。推論時間は約2倍になるが、精度は2倍以上になった。

## 第8章：実機で回す ── プロンプトの1語が推論経路を変える

記事で書いた改善は、すべて実機ベンチマークで検証した。その過程で、記事執筆時には見えなかった新しい問題が3つ浮上した。

### 「ステップ」がシミュレーションを誘発する

追いつき算をベンチマークに追加して回したところ、モデルが96ステップの逐次シミュレーションに突入した。

```
Step 1: t=1, John=4km, Mike=0km. State: [4, 0]
Step 2: t=2, John=8km, Mike=6km. State: [8, 6]
Step 3: t=3, John=12km, Mike=12km. State: [12, 12]
...
Step 96: ...（文脈窓オーバーフロー）
```

「After EACH step, write State」と書くと、小モデルは「step = シミュレーション上の各時刻」と解釈する。追いつき算は方程式 `4t = 6(t-1)` で一発で解けるのに、1時間ずつ追いかけ始めてしまう。

```
修正前: "Then solve step by step. After EACH step, write State: [current values]."
修正後: "Solve using equations when possible. Write State: [values] after each reasoning step."
```

「step by step」を「equations when possible」に変えた。たった3語の変更で、モデルは方程式を立てるようになり、96ステップのシミュレーションは消えた。

### +1変換エラー ── 正しく解いて間違える

方程式を立てるようになっても、新しい失敗パターンが現れた。

```
[Solve]
  Let t = time after Mike starts
  4(t+1) = 6t → t = 2         ← 代数は完璧
  Answer: 2 hours after John left  ← 変換を忘れている！
```

t=2 は「Mike出発から2時間」であり、質問の「John出発から何時間後」に答えるには 1+2=3 とすべきだ。方程式も代数も正しいのに、**最後の1行で変数の意味を取り違える**。

5回テストした結果、同じ問題を3つのパターンで間違えた。

| パターン | 方程式 | 代数 | 解釈 | 最終答 |
|:---------|:-------|:-----|:-----|:------:|
| 正解 | 4t = 6(t-1) | t=3 | 正しい | **3** ✓ |
| +1忘れ | 4(t+1) = 6t | t=2 | John出発からと誤報告 | 2 ✗ |
| 方程式逆 | 6t = 4(t-1) | t=-2 | 追いつけないと結論 | -2 ✗ |

**同じプロンプトで毎回違う間違い方をする。** これは記事冒頭で書いた「5回聞くと5回とも違う間違い」の再現だ。

対策として、Verify段階のプロンプトに1文追加した。

```
修正前: "Check this draft for errors."
修正後: "Check this draft for errors. Verify the final answer directly addresses what the question asked."
```

「最終回答が質問に直接答えているか確認せよ」──この1文で、Verifyが「質問はJohn出発からだが、回答はMike出発からの時間になっている」と気づける可能性を作った。

### 安全弁のジレンマ

しかし、Verifyが+1を修正して「3」に直しても、安全弁がそれを却下していた。

```swift
// 旧ロジック
let answerChanged = solveAnswer != verifyAnswer
if answerChanged { finalRaw = solveRaw }  // Solveの"2"に戻る
```

この安全弁は「Verifyが正解を壊すのを防ぐ」ために導入した。だが同時に「Verifyが誤答を直すのも防ぐ」副作用があった。

Verifyは現在「レビューモード」（既存の回答をチェックして修正する）で動いている。ゼロから再解するのではなく、Solveの出力を見てエラーだけを直す方式だ。この方式なら、Verifyの修正を信頼してよい。

```swift
// 新ロジック
if isRefusal { finalRaw = solveRaw }  // 拒否時のみSolveを使う
else { finalRaw = verifyRaw }         // それ以外はVerifyを信頼
```

### 再テスト結果

修正後、追いつき算を含むベンチマークを実機で実行した。

| 問題 | 期待 | 修正前 | 修正後 |
|:-----|:----:|:------:|:------:|
| cond-5step | 7 | 7 ✓ | 7 ✓ |
| evenodd-6 | 7 | 7 ✓ | 7 ✓ |
| 2var | 21 | 21 ✓ | 21 ✓ |
| price-4step | $56 | $56 ✓ | $56 ✓ |
| **catchup (EN)** | **3** | **—** | **3 ✓** |
| **catchup (JP)** | **3** | **—** | **3 ✓** |

既存問題にリグレッションなし。追いつき算は英語・日本語ともに正解を出せるようになった。ただし非決定性は残っており、**常に正解するとは限らない**。3回中2回正解、1回は方程式を逆に立てるような結果になる。

これが小さなモデルの現実だ。100%の精度は得られない。だが、40%を70%以上に引き上げることはできた。

## 第9章：壁 ── パイプラインでは超えられないもの

ここまで成功の話をしてきた。だが、正直に失敗も記録しておく。

### ハルシネーション

実在する人物について聞くと、モデルは自信満々にデタラメを語る。

```
Q: "柑野みかについて教えて"
A: "柑野みかは日本の女性ファッションデザイナーで..."  ← 完全な作り話
```

Web検索を組み合わせれば正しい情報を取得できる場合もある。だが、検索結果が見つからなかった時、モデルは「わかりません」と言う代わりに物語を創作する。

「知らないなら正直に言え」というプロンプトを追加しても、改善は限定的だった。**小さなモデルは「自分が知らないこと」を知ることが、最も苦手**なのだ。

### 概念の壁

カップのスワップ問題では、全パイプラインが同じ間違いをした。

```
Q: "コインがカップAの下。A⇔C交換、B⇔D交換（両方空）、A⇔D交換、B⇔C交換"
正解: B    全パイプラインの回答: C
```

「空のカップ同士を交換しても何も起きない」──この概念をモデルが理解していない。何回独立にセッションを立てても、同じ概念的エラーを繰り返す。

**パイプラインで修正できるのは「ランダムなエラー」（計算ミス、数え間違い）だ。モデルが概念自体を持っていない「体系的エラー」には無力。**

## エピローグ：小さなAIの正しい使い方

この実験で得られた7つの原則を記す。

### 1. 層は最小限に
ステップを増やすほど精度は落ちる。2段階がスイートスポット。

### 2. セッションは独立に
マルチターンはコンテキスト汚染を起こす。各段階に新しいセッションを。

### 3. 検証は「他人の目」で
自己批評は確証バイアスに陥る。検証者には問題と結論だけを渡せ。

### 4. 「何で考えるか」を先に決めさせろ
「考えろ」では漠然としすぎる。「アプローチを特定せよ」で正しい道具が選ばれる。

### 5. 「ステップ」の解釈を制御しろ
「step by step」は小モデルにシミュレーションを誘発する。「方程式を使え」で推論経路が変わる。

### 6. 検証者の修正を信頼しろ
安全弁は「正解を守る」と同時に「修正を潰す」。レビューモードなら検証者を信頼すべきだ。

### 7. 限界を知れ
ランダムエラーは直せる。体系的エラーは直せない。知識の壁はモデルサイズの壁だ。そして非決定性は消えない──同じ問題を3回解けば、3回のうち1回は違う間違い方をする。

---

Foundation Modelsは、ChatGPTの代替にはならない。知識量も、創造性も、言語の豊かさも、桁が違う。

だが、**「正しく考える力」は、外部フレームワークである程度補える。** 2段階の独立セッションと、たった数語のプロンプト修正で、追いつき算が解けるようになった。

そして何より面白かったのは、実験が**直感に反する結果の連続**だったことだ。

- ステップを増やせば賢くなる → **逆。エラーが蓄積する**
- 自分で見直せば精度が上がる → **逆。確証バイアスに陥る**
- 多数決で安定する → **逆。同じ間違いを3回繰り返す**
- 丁寧に手順を踏ませれば良い → **逆。シミュレーション暴走する**
- 安全弁は正解を守る → **半分正しく、半分間違い。修正も潰す**

正解は、最もシンプルな構造の中にあった。

**「一度解いて、別の自分にチェックしてもらう。」**

論文の査読、コードレビュー、ダブルチェック──人間が何百年も前から実践してきた知恵が、3Bパラメータの小さなAIにもそのまま通用した。

小さなAIは、人間と同じ方法で、少しだけ賢くなれる。
